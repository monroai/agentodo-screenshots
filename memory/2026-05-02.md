### Root Cause of Most Failures
**"Publishing changes from background threads is not allowed"** — this SwiftUI runtime warning appeared across every failing test suite. `@Published` or `@Observable` properties being mutated off the main thread. This single threading violation is likely cascading into:
- App crashes mid-test (PID mismatch)
- Bare key handler not firing (event monitor on wrong thread)
- Checkbox toggles causing race conditions
- CleanupManager not holding completed rows visible

### Key Technical Discoveries

**Pattern #9 is everywhere:** `_id`, `_title`, etc. use optional backing stores with `@Transient` fallbacks because SwiftData on macOS 26 transiently wipes values during cascade operations. This creates a whole class of "flicker" bugs (title shows "Untitled" for one frame, editor loses stableLocationId, etc.). The app has mitigations but they're fragile — every new view that reads these properties needs its own `lastKnown*` cache.

**Swift 6 + XCTest is broken:** The UI test base class uses `@MainActor` but `XCTestCase.setUpWithError()` is `nonisolated` in Swift 6. Every `XCTContext.runActivity` closure also triggers sendability errors. Fix: set UI test target to `SWIFT_VERSION: 5.0` + `SWIFT_STRICT_CONCURRENCY: minimal` in project.yml.

**Signing cert expired:** "Apple Development: Monro AI (F38256T2W2)" was invalid. Oleksandr fixed it mid-session. Ad-hoc signing (`CODE_SIGN_IDENTITY=-`) works for building but NOT for running UI tests (entitlements require real signing).

**Location features are architecturally ambitious but half-baked:**
- CLMonitor crashes on macOS with WhenInUse auth (needs Always)
- Cross-device dedup has a self-matching bug that eats departure notifications
- 20-region CLMonitor limit with no clear user feedback when exceeded
- LocationIntelligenceService nightly clustering runs on @MainActor with O(n²)
- "Add point at my location" permanently disabled on desktop Macs without GPS

**Attachment system has a marker problem:** Notes store raw `{{att:UUID:storedName}}` markers that are only rendered by the MarkdownEditor. Any other context (CLI, widget, export, clipboard) shows the raw markers. No fallback rendering.

### Lessons Learned (New)
1. **Run the existing tests first.** The app had a full XCUITest suite that was failing to compile AND had 76% failure rate. Running these tests early would have identified the background-thread issue much sooner.
2. **Code review catches more than screenshots.** On a Mac mini without a display/mouse, deep code review found issues I never would have caught visually — threading violations, race conditions, dedup logic bugs, missing validation.
3. **Swift 6 strict concurrency is a minefield for tests.** Production code can be fully Swift 6, but XCTest + @MainActor don't mix. Keep test targets in Swift 5 mode.
4. **"Pattern #9" (SwiftData optional backing stores) is the project's Achilles heel.** Every model, every view, every service has workarounds for this. It's a systemic issue that needs a systemic fix (or a SwiftData version update that resolves it).
5. **File size limits are easy to forget.** The attachment system has no size cap. Same for OmniFocus imports (no cancel). Always ask: "what if the input is 100x larger than expected?"
6. **Batch filing with subagents is efficient.** Filing 20 issues per subagent, with pre-written descriptions, takes ~3 minutes per batch. Much faster than filing one at a time.
7. **The signing certificate expires/revokes periodically.** Always check `security find-identity -v -p codesigning` before starting a test session that needs real signing.

### Architecture Notes

- **DEVELOPER_DIR** must point to `/Applications/Xcode.app/Contents/Developer` (not CommandLineTools)
- **Scheme name** is "AgenToDo" (capital T D), not "AgentoDo"
- **DerivedData app path:** `~/Library/Developer/Xcode/DerivedData/AgenToDo-*/Build/Products/Debug/AgentToDo.app`
- **xcresulttool** at `/Applications/Xcode.app/Contents/Developer/usr/bin/xcresulttool`
